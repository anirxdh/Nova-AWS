**Approach:** Use Groq Whisper's existing batch API but with a WebSocket endpoint that accepts audio chunks from the extension during recording. The backend accumulates chunks and begins transcription immediately when the WebSocket closes (user releases key). This gives a head start because:
1. Audio data is already on the server when the user releases the key (no upload delay)
2. The "Transcribing..." state can begin earlier

This is a pragmatic streaming approach -- true real-time partial transcripts would require a different STT service, but eliminating the upload latency is the biggest win.

**In `backend/services/nova_sonic.py`:**

Add a new function `transcribe_audio_streaming()` with the same signature as `transcribe_audio`. This function is identical to `transcribe_audio` but is designed to be called from the WebSocket handler where audio bytes are already accumulated server-side. Keep it as a separate function so the batch endpoint is untouched.

```python
def transcribe_audio_streaming(audio_chunks: list[bytes], mime_type: str) -> str:
    """Transcribe pre-accumulated audio chunks using Groq Whisper."""
    audio_bytes = b"".join(audio_chunks)
    return transcribe_audio(audio_bytes, mime_type)
```

**In `backend/routers/transcribe.py`:**

Add a WebSocket endpoint alongside the existing POST endpoint. Import `WebSocket` and `WebSocketDisconnect` from fastapi.

```python
@router.websocket("/transcribe/stream")
async def transcribe_stream(ws: WebSocket):
```

This endpoint:
1. Accepts the WebSocket connection.
2. Receives a JSON text message first: `{"mime_type": "audio/webm"}` to configure.
3. Accumulates binary messages (audio chunks) into a `list[bytes]`.
4. When the client closes the WebSocket (or sends a text message `{"action": "done"}`):
   - Emits SSE event `{"stage": "transcribing"}` via event_bus.
   - Calls `transcribe_audio_streaming(chunks, mime_type)` via `asyncio.to_thread()`.
   - Sends back a text message with `{"transcript": "..."}` before closing.
   - Emits SSE event `{"stage": "done", "transcript": transcript}`.
5. Error handling: on any exception, sends `{"error": "..."}` and closes.

Import `transcribe_audio_streaming` from `backend.services.nova_sonic`.

Keep the existing `POST /transcribe` endpoint completely untouched -- it remains the batch fallback.

**In `src/background/api/backend-client.ts`:**

Add a new function `transcribeAudioStreaming()`:
```typescript
export async function transcribeAudioStreaming(
  audioBase64: string,
  mimeType: string
): Promise<string> {
```

This function:
1. Opens a WebSocket to `ws://localhost:8000/transcribe/stream`.
2. Sends config message: `JSON.stringify({ mime_type: mimeType })`.
3. Converts base64 to binary (same as existing `transcribeAudio` logic), then sends as a single binary message (the full recording). Even though the WebSocket supports chunking, sending the full recording at once still eliminates the HTTP overhead and multipart encoding delay.
4. Sends done signal: `JSON.stringify({ action: "done" })`.
5. Waits for the transcript response message: parses JSON, returns `data.transcript`.
6. On WebSocket error or timeout (5s): throws an error so the caller can fall back to batch mode.
7. Closes the WebSocket.

Wrap the WebSocket interaction in a `Promise` with a 15-second timeout. Use `new WebSocket(url)` (available in service worker context).
