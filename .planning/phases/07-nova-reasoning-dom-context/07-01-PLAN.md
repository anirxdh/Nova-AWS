<objective>
Create the Nova 2 Lite reasoning service and POST /task API endpoint that receives a user command, screenshot, and DOM snapshot, then returns either a text answer or structured action steps.

Purpose: This is the brain of the agent -- Nova 2 Lite interprets what the user wants by seeing the page (screenshot + DOM) and hearing the command. Without this, the extension can transcribe speech but cannot understand or act on it.

Output: POST /task endpoint returning `{ type: "answer", text }` or `{ type: "steps", actions: [...] }`, backed by Nova 2 Lite via boto3.

  <name>Task 1: Create Nova 2 Lite reasoning service</name>
  <files>backend/services/nova_reasoning.py</files>
  <action>
Create `backend/services/nova_reasoning.py` with a `reason_about_page(command: str, screenshot_base64: str, dom_snapshot: dict) -> dict` function that:

1. Reads AWS credentials from env (same pattern as nova_sonic.py -- os.getenv for AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_REGION)
2. Creates a boto3 bedrock-runtime client
3. Calls `client.converse` (NOT converse_stream -- we want the full response, not streaming) with model ID `us.amazon.nova-lite-v1:0`
4. Builds the message with a system prompt and a user message containing:
   - The screenshot as a base64 image block (use the Bedrock converse image format: `{"image": {"format": "png", "source": {"bytes": base64_decoded_bytes}}}`)
   - The DOM snapshot as a text block (JSON-serialized)
   - The user command as a text block

System prompt (define as a module-level constant SYSTEM_PROMPT):
```
You are ScreenSense, a screen-aware AI assistant embedded in a Chrome extension. You receive:
1. A screenshot of the user's current browser tab
2. A DOM snapshot with interactive elements (buttons, links, inputs, forms) and their CSS selectors
3. A voice command from the user

Your job:
- If the user is asking a QUESTION about what they see, respond with a JSON object: {"type": "answer", "text": "your answer here"}
- If the user is giving a TASK command (e.g., "click add to cart", "search for headphones"), respond with a JSON object: {"type": "steps", "actions": [{"action": "click", "selector": "#add-to-cart", "description": "Click the Add to Cart button"}, ...]}

Supported action types: click, type, navigate, scroll
- click: {"action": "click", "selector": "CSS selector", "description": "what this does"}
- type: {"action": "type", "selector": "CSS selector", "value": "text to type", "description": "what this does"}
- navigate: {"action": "navigate", "url": "URL to navigate to", "description": "what this does"}
- scroll: {"action": "scroll", "direction": "up|down", "description": "what this does"}

Use the DOM snapshot selectors when available. Respond ONLY with the JSON object, no markdown, no explanation.
```

5. Parses the response text from `response["output"]["message"]["content"][0]["text"]`
6. Attempts `json.loads()` on the response text. If it parses as a dict with a `type` field ("answer" or "steps"), return it. If JSON parsing fails, wrap as `{"type": "answer", "text": response_text}` (fallback for when Nova returns plain text).
7. Error handling follows the same pattern as nova_sonic.py: catch NoCredentialsError, PartialCredentialsError, ClientError with descriptive re-raises as ValueError.

Note: Use `client.converse` (synchronous, not streaming) since we need the complete JSON response before we can determine answer vs steps. The function is sync (boto3 is sync) -- the router will wrap it with asyncio.to_thread().

The screenshot_base64 parameter is a base64-encoded PNG string (no data:image/png;base64, prefix -- just the raw base64). Decode it with `base64.b64decode(screenshot_base64)` to get bytes for the Bedrock image block.

   ```python
   class TaskRequest(BaseModel):

       screenshot: str       # Base64-encoded PNG screenshot
       dom_snapshot: dict    # Structured DOM data (buttons, links, inputs, etc.)
   ```

2. Create the router and endpoint:
   ```python
   router = APIRouter()

   @router.post("/task")
   async def process_task(request: TaskRequest):
   ```

3. The endpoint should:
   - Emit SSE event: `await event_bus.emit("status", {"stage": "understanding"})` at start

   - Return the result dict directly (FastAPI auto-serializes)
   - Error handling: catch ValueError (credentials/AWS errors) -> 500, catch Exception -> 500, emit error SSE events like /transcribe does
